# Management OS - Unified Reference Guide\n\n## Executive Summary\n\n**Company**: LOJIK for Professional Property Appraisers Inc  \n**System**: React/Supabase property appraisal workflow management platform  \n**Development**: 100% online via GitHub Codespaces (no local files)  \n**Deployment**: GitHub → Vercel  \n**Tech Stack**: React, Supabase PostgreSQL, file processing pipelines  \n**Scale**: Production system handling 50-100K+ property records\n\n**Mission**: Transform Excel-based workflows into database-driven intelligence. Handle entire lifecycle from job creation to appeal defense with sophisticated data processing, real-time analytics, and workflow orchestration.\n\n**Key Value Propositions:**\n- Transforms Jim's methodology into documented, repeatable processes\n- Enables knowledge transfer and team standardization\n- Provides complete audit trails for legal defense\n- Scales property assessment operations to enterprise level\n- Eliminates Excel hell and Google Drive chaos\n\n## Recent Schema Optimizations (September 2024)\n\n### Job Archive & Lifecycle Management (January 2025)\n- **New Feature**: Archive/restore jobs with checklist validation\n- **Database Fields**: Added `archived_at` (timestamp) and `archived_by` (uuid) to `jobs` table\n- **Billing Integration**: Archived jobs automatically move from Active to Legacy in BillingManagement\n- **User Experience**: Warning modal if checklist incomplete, option to archive anyway, restore button on archived jobs\n- **Filter Logic**: `activeJobs` excludes archived, `legacyJobs` includes archived regardless of job_type\n- **Data Preservation**: All payment history and job data preserved when archived\n\n### The Performance Crisis\n- **Problem**: 50,000+ total properties across all jobs (16,000+ in largest single job)\n- **Issue**: Each property storing its own `raw_data` JSONB field created massive bottlenecks\n- **Symptoms**: File uploads stuck in infinite initialization loops, protected field fetching difficulties\n\n### The Solution Architecture\n**Consolidated Raw Data Storage Pattern:**\n- **Before**: Every property stored complete `raw_data` JSONB (50,000+ copies!)\n- **After**: Single `raw_file_content` TEXT field at job level (1 copy instead of 16,000!)\n- **Result**: Dramatically reduced database size, eliminated initialization loops, faster queries\n\n**Field Migration Strategy:**\nThe optimization split `property_records` into THREE destinations:\n\n1. **jobs table** - Got job-wide settings:\n   - `project_start_date` (applies to all properties in job)\n   - `validation_status` (overall job validation)\n   - `raw_file_content` (consolidated raw data for entire job)\n   - Added tracking: `raw_file_parsed_at`, `raw_file_rows_count`, `raw_file_size`\n   - `external_inspectors` (comma-separated list of client codes)\n\n2. **property_market_analysis** (NEW TABLE) - Got property-specific analysis fields:\n   - Manual entry fields: `asset_key_page`, `asset_map_page`, `asset_zoning`, `location_analysis`, `new_vcs`\n   - Calculated values: `values_norm_size`, `values_norm_time`\n   - `sales_history` tracking\n   - Property-level `validation_status`\n\n3. **market_land_valuation** - Enhanced with:\n   - Granular economic obsolescence tracking (4 eco_obs_* fields)\n   - Consolidated `zoning_config` JSONB\n   - New `target_allocation` and `ignored_issues` features\n\n### Performance Benefits\n- Database size reduced by ~95% for raw data storage\n- Query performance improved 10x for property listings\n- Eliminated timeout issues during file uploads\n- Better separation of concerns (job vs property vs analysis data)\n- Validation override syncing prevents duplicate key errors\n- Processing modal batches decisions for single database write\n- Override records include complete property data to avoid lookups\n\n## System Architecture\n\n### Core Data Flow Pattern\n\n```\nVendor Files (BRT/Microsystems) → FileUploadButton (Comparison) → Processors/Updaters → Database → Analytics → UI\n```\n\n**Critical Pattern: INSERT vs UPSERT**\n- **Initial Import**: AdminJobManagement ��� processors → INSERT new records (job creation)\n- **Ongoing Updates**: FileUploadButton → updaters → UPSERT operations (file maintenance)\n\n### Module Data Loading Architecture (NEW)\n\n**JobContainer as Central Data Orchestrator**:\n```\nJobContainer (loads once with pagination)\n    ├── Loads ALL property_records (handles 5K-16K+ records efficiently)\n    ├── Applies assignment filtering if has_property_assignments = true\n    ├── Fetches job metadata (code definitions, vendor type, dates)\n    ├── Shows progress in banner (not modal!)\n    └── Distributes via props to:\n        ├── ProductionTracker (existing pattern)\n        ├── MarketLandAnalysis (NEW: no double loading!)\n        ├── ManagementChecklist  \n        ├── FinalValuation\n        └── AppealCoverage\n```\n\n**Assignment-Aware Loading Pattern**:\n- Checks `jobs.has_property_assignments` flag\n- When true, adds `.eq('is_assigned_property', true)` to query\n- Shows "Assigned Properties Only" badge in UI banner\n- Single query with proper filtering at database level\n- Consistent across all modules receiving data\n\n**Props Interface for ProductionTracker**:\n- `properties`: Array of current property_records (filtered if has_property_assignments)\n- `inspectionData`: Array of inspection_data records for the job\n- `employees`: Array of all employees for inspector validation\n- `dataUpdateNotification`: Signal from JobContainer when data refreshes\n- `clearDataNotification`: Function to clear the notification\n- `latestFileVersion`: Current file version number\n- `onUpdateWorkflowStats`: Callback to update App.js state\n- `currentWorkflowStats`: Current workflow statistics from App.js\n\n**Loading Performance Metrics**:\n- 5,000 properties: ~5 seconds\n- 13,000 properties: ~13 seconds\n- 16,000+ properties: ~16-20 seconds\n- Batch size: 1000 records per page\n- Small delay between batches: 100ms\n\n### Repository Structure\n\n**Complete
